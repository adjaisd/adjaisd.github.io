编译与执行问题

 2023年7月中旬，在学习野火的教程时，由于系统镜像版本不一致，gcc版本不一致，GLIBC支持的版本也不一次，编译时出现GLIBC_2.34 not found问题;

 8月初，项目中用到语音唤醒，在编译和运行时出错，可以编译，但运行失败提示libmsc.so not found，问题解决后迁移到Cmake中，问题又出现了，经过反复试验问题解决;

 8月15包括前两天，采用Clion分别配合cygwin、mingw-win32-seh、mingw-posix-seh等版本时，当我点击Clion右上角的运行时，会报错”Error running ‘xxx’: File not found: xxx”，但在cmd中能正常运行，解决过程在第3部分.

交叉编译”GLIBC_2.34 not found”

 gcc符号链接的原二进制文件比如x86_64-linux-gnu-gcc-7、arm-linux-gnueabihf-gcc-8、arm-none-eabi-gcc等;

 安装其他平台的gcc工具链: apt install gcc-arm-linux-gnueabihf，也可在Linaro官网下载;

 编译依赖版本问题参考该文章，也可参考该文章;

 说明下环境，本地采用Ubuntu22.04镜像搭建，工具链arm-linux-gnueabi-gcc版本为11.4.0，目标机器为野火imx6ull，交叉编译后在目标执行报错'GLIBC_2.34' not found，解决如下:

 通过ldd命令打印目标机器依赖的共享库;

root@npi:~# arm-linux-gnueabihf-gcc hello.c -o hello 
root@npi:~# ldd hello
    linux-vdso.so.1 (0x7ee4f000)
    libc.so.6 => /lib/arm-linux-gnueabihf/libc.so.6 (0x76dfb000)
    /lib/ld-linux-armhf.so.3 (0x76f07000)

 发现3个库文件，通过find指令查找各自位置，发现只有两个依赖，且在/lib/arm-linux-gnueabihf目录下;

# 查找依赖库的位置
root@npi:~# find / -name linux-vdso.so.1
root@npi:~# find / -name libc.so.6
/lib/libc.so.6
/lib/arm-linux-gnueabihf/libc.so.6
root@npi:~# find / -name ld-linux-armhf.so.3
/lib/ld-linux-armhf.so.3
/lib/arm-linux-gnueabihf/ld-linux-armhf.so.3
# 查看各原文件具体信息
root@npi:~# ls -l /lib/libc.so.6 /lib/ld-linux-armhf.so.3 
lrwxrwxrwx 1 root   root        30 Mar 16  2022 /lib/ld-linux-armhf.so.3 -> arm-linux-gnueabihf/ld-2.28.so
-rwxr-xr-x 1 debian debian 1296004 Nov 28  2021 /lib/libc.so.6
root@npi:~# ls /lib/arm-linux-gnueabihf/libc.so.6 /lib/arm-linux-gnueabihf/libc-2.28.so -l
-rwxr-xr-x 1 root root 952028 Mar 16  2022 /lib/arm-linux-gnueabihf/libc-2.28.so
lrwxrwxrwx 1 root root     12 Mar 16  2022 /lib/arm-linux-gnueabihf/libc.so.6 -> libc-2.28.so

 由于gcc版本不同，将/lib/arm-linux-gnueabihf目录下的libc于ld文件拷贝到本地虚拟机进行交叉编译;

root@ubuntu:~# mkdir -p lib
root@ubuntu:~# scp npi:/lib/arm-linux-gnueabihf/libc.so.6 ./lib/
libc.so.6                                    100% 1266KB   2.2MB/s   00:00    
root@ubuntu:~# scp npi:/lib/arm-linux-gnueabihf/ld-linux-armhf.so.3 ./lib/
ld-linux-armhf.so.3                          100%  103KB   1.7MB/s   00:00

 交叉编译需参考该文章，该文章通过glibc-all-in-one下载的库进行编译，执行update_list后，发现list目录下可能不存在所需的库，另外list文件起名有些疑惑，因此，后续通过拷贝目标机器的依赖库进行编译;

root@ubuntu:~# name="arm-hello"
root@ubuntu:~# arm-linux-gnueabihf-gcc hello.c -o ${name} ./lib/*
root@ubuntu:~# scp ${name} npi:/root/
arm-hello                                                100% 7808   655.8KB/s   00:00    
root@ubuntu:~# ssh npi /root/${name}
Hello, world!

配置文件方式

 后来做语音相关的项目，采用Cmake工具，在build前，通过Cmake的set(ENV{LD_LIBRARY_PATH} ${LIB_PATH})指定链接路径的环境变量，编译成功后通过ldd发现依赖库libmsc.so是not found，

root@ubuntu:~/workspace/STM32-learning/build/100/x64/bin# ldd awaken 
        linux-vdso.so.1 (0x00007ffcd6fb8000)
        libmsc.so => not found
        libasound.so.2 => /lib/x86_64-linux-gnu/libasound.so.2 (0x00007f1414273000)
        libc.so.6 => /lib/x86_64-linux-gnu/libc.so.6 (0x00007f1414000000)
        /lib64/ld-linux-x86-64.so.2 (0x00007f141438e000)

 尝试通过绝对路径的方式，target_link_libraries(awaken1 "${LIB_PATH}/libmsc.so" "${LIB_PATH}/libw_ivw.so")，发现当Cmake采用多层嵌套即多层add_subjectory后，采用get_filename_component设置LIB_PATH为绝对路径也无效，产生的link.txt文件如下所示:

/usr/bin/gcc -m32 -g CMakeFiles/awaken.dir/src/awakenMain.c.obj -o ../../../../../../bin/awaken   -L/root/workspace/STM32-learning/lib/x86_64-linux-gnu  ../../../../../../../../../lib/x86-linux-gnu/libmsc.so ../../../../../../../../../lib/x86-linux-gnu/libw_ivw.so -lasound -lc

 接着，采用配置/etc/ld.so.conf或者在 /etc/ld.so.conf.d目录下自定义配置文件的方式来解决;

root@ubuntu:~/workspace/STM32-learning/build/100/x64/bin# cat /etc/ld.so.conf.d/STM32-learning.conf 
/root/workspace/STM32-learning/lib/x86_64-linux-gnu
root@ubuntu:~/workspace/STM32-learning/build/100/x64/bin# ldconfig

 关于修改LD搜索路径(LIBRARY_PATH)，也可在/etc/profile.d目录下新建sh文件，如下:

root@ubuntu:~# cat > /etc/profile.d/ld.sh << EOF
export LD_LIBRARY_PATH=/root/workspace/STM32-learning/lib/x86_64-linux-gnu:\$LD_LIBRARY_PATH
EOF
root@ubuntu:~# source /etc/profile

库重名问题

 当ld的搜索路径，存在重命名的动态库时，可采用绝对路径的方式，如果采用Cmake且多层嵌套的话，link.txt中不会转换为绝对路径，不一定适用;

 由于没接触过动态库重名问题，可参考改文章.

执行问题”Error running ‘xxx’: File not found: xxx”

 在Linux系统中执行target可以顺利运行，然而，在Windows平台，文件存在且能在cmd中执行，但在Clion中始终报错;

 Clion中打开新项目，采用同样的toolchain与同样的Cmake配置，target可顺利执行;

 我删掉了出错的项目，克隆并重新载入，构建并运行，但仍是相同的错误，事情又回到了起点;

 清理了build目录，重新构建，项目采用的是”ninja generator”，我注意到正常的项目下不存在CmakeError.log，而出错的项目出现了CmakeError.log文件，关键报错如下:

Cannot copy output executable
  ''
to destination specified by COPY_FILE:
  'D:/xxx/build/local/bundled/CMakeFiles/3.25.2/CMakeDetermineCompilerABI_CXX.bin'
Recorded try_compile output location doesn't exist:
  D:/xxx/build/local/bundled/CMakeFiles/CMakeScratch/TryCompile-7wb0bl/cmTC_8a406

 查找了相关issue，问题是Cmake的版本3.25出问题了，需要降到3.24版本，我通过cygwin安装了3.23.2版本的cmake，问题可以解决;

 事情并没有结束，事实上我最开始的解决该问题没有参照该issue，因为本机两个同样配置的项目，一个在Clion中可顺利运行，另一个则不能(但在外部cmd可以成功运行);

 在维持Cmake版本3.25.2的情况下，网上几乎没有任何的解决方法，在之后的反复尝试中，我观察到能正常运行的项目它所执行的target名称是/path/to/xxx.exe，而出问题始终报错Error running 'xxx': File not found: /path/to/xxx;

 接着在cmd中尝试了以下指令:

C:\Users\uname>if exist D:\path\to\xxx.exe (echo 1)
1
C:\Users\uname>if exist D:\path\to\xxx (echo 1)

 渐渐确定，虽然D:\path\to\xxx可执行，但不代表该二进制文件存在，也许Cmake版本3.25.2中对文件存在的判断没有考虑到后缀.exe，于是出现了File not found;

 因此问题可定位到Cmake的配置文件中，在Clion对该CmakeLists.txt进行Debug，察觉到即便配置了toolchain，但存在CMAKE_EXECUTABLE_FORMAT=Unknown，搜索了Cmake对target的后缀的配置方法，找到了CMAKE_EXECUTABLE_SUFFIX参数，执行Cmake指令时可指定-DCMAKE_EXECUTABLE_SUFFIX=.exe，相关的Cmake配置参数可参考该文章;

 经过以上分析，无需修改Cmake版本，解决了Clion运行target报错Error running 'xxx': File not found: /path/to/xxx的问题.
