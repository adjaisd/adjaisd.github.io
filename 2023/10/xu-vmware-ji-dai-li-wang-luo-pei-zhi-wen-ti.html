续VMware及代理网络配置问题

 由于最近在学汇编、Linux和操作系统更底层的知识，发现很多资料或视频基于VScode环境，而采用Clion开发的较少，因此琢磨了下VScode的使用，在这过程中遇到了远程开发的网络配置相关问题;

 此外，最近发生了一件非常有争议的事，承德一程序员远程工作而被罚没3年收入，为此，有必要了解下代理软件的注意事项；

 接下来会分享自己使用VMware及代理软件的使用技巧.

VMware网络配置

 首先声明下，宿主机为win11，VMware创建虚拟机系统为ubuntu22.04LTS，网上关于我在工作学习上遇到的问题的解决方案较少，解决方案是在长期使用的过程中逐渐完善的，最后通过shell脚本实现.

需求

 在任何网络下(无网络、桥接网络、动态网络)，都能通过固定ip可以访问虚拟机;

  1. 考虑一个场景，比如在家、公司等比较固定的工作学习场所

 考虑采用bridge网络，网上有很多配置静态ip的教程，然而如果经常切换网络环境(比如在公司和在家之间切换网络)，便需要反复地配置静态ip;

 考虑采用nat网络，尽管nat模式是让虚拟机联网最简单的方式，但远程开发需要麻烦地进行端口映射;

  1. 第二个工作场景，比如在出差、家庭断电等非固定工作场所或网络设备发现异常的情况

 在出差时，或者固定工作场所的网络因断电、宽带欠费等而断网，大多依靠手机热点来联网;

 如果配置为bridge网络，由于热点的网段是不固定的，尽管可通过特殊方式为手机配置特定网段的dhcp，但这有时需要获取root权限(可参考此前写的手机初次刷机文章)，经历非常麻烦的步骤后才能解锁root，因此不推荐该方法.

 如果配置为nat模式并通过dhcp分配ip，可以实现虚拟机和宿主机都能联网，但仍需端口映射才能远程开发，由于dhcp分配的ip可能会发生变化，端口映射也需要相应调整;

  1. 不限制外部网络环境

 从以上两点的分析中，我们发现使用单纯地采用nat网络或者bridge网络都无法从容应对各种场景，并且如果不考虑使用特定设备或针对性配置的话，两者网络模式都无法实现较方便地访问固定ip(bridge模式采用静态ip会由于工作场所切换(公司、学校、家庭路由器的网段不一致)而频繁更改配置，而nat模式在宿主机需访问多个虚拟机的服务时，需手动配置端口映射);

 排除掉bridge和nat模式，还剩host模式，由于host模式类似bridge模式，宿主机相当于路由器，虚拟机相当于局域网中的设备，尽管在host模式下，虚拟机不能通过宿主机访问互联网，但也实现了任何网络环境下都能与宿主机直连;

 考虑到虚拟机联网需求，可以采用多网卡配置方式，比如host+bridge、host+nat或者host+bridge+nat等组合;

  • host+bridge双网卡

     在该模式下，几乎可以适用于大多数固定的场景了(比如在家或在公司)，宿主机通过host远程登录虚拟机，虚拟机通过bridge网络能访问到互联网;

     该方法的优点在于: 如果路由器出现了问题(比如断电、宽带欠费等)，那么虚拟机便无法访问互联网，但由于存在host网络，宿主机还是能与虚拟机直接交互的，如果需要联网的话，通过手机热点，虚拟机的bridge网卡应设为dhcp模式;

     然而该方法存在几个问题，有些软件需通过代理才能访问(除非路由器或热点配置了代理、或者在虚拟机内部配置http_proxy)，如果没有这些，从头配置比较麻烦(有的路由器不一定支持，需要对路由器刷机，虚拟机需配置网络);

  • host+nat双网卡

     为了解决host+bridge组合存在的问题(这里不考虑修改外部网络或修改虚拟机内部网络proxy配置);

     通过host+nat组合，宿主机再配合开启tun模式的clash软件，虚拟机内部流量可被成功分流;

     尽管上述host+bridge和host+nat两种模式可以应对大部分情况，前者适用于需要联网但不需要代理的场景，而后者适用于需要联网且需要流量被代理的场景，如果要经常在两者之间切换的话，还是不太灵活;

  • host+bridge+nat三网卡

     针对上述问题，琢磨过三模式的网卡配置，但发现虚拟机全局流量走的都是nat网络，起初猜测为路由，但在ping时发现目标为宿主机clash开启tun模式的网卡ip，然后猜测是存在dns缓存(清楚dns缓存可通过resolvectl指令，配置/etc/resolve.conf文件不一定有用，没必要安装其他dns工具)，经过尝试，虚拟机果然不走nat网络了(域名解析为真实的ip而非fakeip)，然而一旦存在dns解析为fakeip，便会添加相应的路由，之后的流量也都会走nat网络;

     尽管可以配置成功三网卡，但如果宿主机开启clash的tun模式代理，那么虚拟机最后的流量总会走nat网络，当然也可能我的路由配置存在问题;

  • host+bridge网卡和host+nat网卡

     为了灵活地切换，虚拟机要有三网卡，分别对应三种网络模式;

     一旦需要在host+bridge(简称组合1)和host+nat(简称组合2)这两种组合中切换，可以编写脚本然后一条指令切换了;

实现

  1. 准备工作

 再次声明下宿主机win11，虚拟机为ubuntu22.04，宿主机安装clash开启tun模式的网卡，如果配置不一样可作为参考;

  1. 编写脚本

 完成VMware网卡配置后，需要为虚拟机添加三个网卡，便于之后切换各种网卡组合，本人的网卡分别为ens33(bridge模式)、ens37(host模式)、ens38(nat模式);

 下面给出手写的脚本update-net.sh(包括开启host+bridge、host+nat组合模式)，在ubuntu22.04LTS-desktop上测试成功:

#!/bin/bash

mode=""    # 网络模式
vm_addr="" # 虚拟机ip
gateway="" # 网关
usage="
usage:
  --mode    : 模式，取值可为'bridge'、'nat'
  --net     : 虚拟机ip
  --gw      : 网关
  --help    : 帮助
examples:
  1. sudo bash update-net.sh --mode bridge --net 192.168.31.100/24 --gw 192.168.31.1
  2. sudo bash update-net.sh --mode nat    --net 192.168.255.100/24 --gw 192.168.255.1
"
options=$(getopt -o f:v --long mode:,net:,gw:,help -- "$@")
eval set -- "$options"
while true; do
  case "$1" in
  --mode) mode="$2" && mode_flag=1 && shift 2 ;;
  --net) net="$2" && net_flag=1 && shift 2 ;;
  --gw) gateway="$2" && gw_flag=1 && shift 2 ;;
  --help) echo "${usage}" && shift 1 && exit ;;
  --) shift && break ;;
  *) echo "没有该选项 $1" && exit 1 ;;
  esac
done
# 检查参数
if [ ! $mode_flag ] || [ ! $net_flag ] || [ ! $gw_flag ]; then
  echo "缺少参数"
  exit
fi

getSubnetIP() { # 获取子网ip
  ip_address=${net%/*}
  subnet_mask_length=${net#*/}
  # 将子网掩码长度转换为32位的二进制表示形式
  subnet_mask="" && for ((i = 0; i < 32; i++)); do
    if ((i < subnet_mask_length)); then
      subnet_mask+="1"
    else subnet_mask+="0"; fi
  done

  # 将IP地址和子网掩码转换为32位的二进制表示形式
  IFS='.' read -r -a ip_parts <<<"$ip_address"
  ip_binary="" && for part in "${ip_parts[@]}"; do
    ip_binary+="$(printf "%08d" "$(bc <<<"obase=2;$part")")"
  done

  subnet_binary="" && for ((i = 0; i < 32; i++)); do
    subnet_binary+="$((${subnet_mask:i:1} & ${ip_binary:i:1}))"
  done
  # 将二进制转换为十进制并获取网络段的IP地址
  subnet_ip="" && for ((i = 0; i < 32; i += 8)); do
    subnet_ip+="$(printf "%d" "$((2#${subnet_binary:i:8}))")."
  done
  subnet_ip="${subnet_ip%?}" # 去除最后一个多余的点号
}

configNetwork() {
  if [ "${mode}" = "bridge" ]; then
    netcard="ens33"
  elif [ "${mode}" = "nat" ]; then
    netcard="ens38"
  else
    echo "mode error"
    exit
  fi
  echo "模式(mode) : ${mode}"
  echo "虚拟机ip(net) : ${net}"
  echo "网关(gw) : ${gateway}"
  echo "子网ip(subnet_ip) : ${subnet_ip}"
  {
    ifconfig ens33 down
    ifconfig ens37 down
    ifconfig ens38 down
  } &
  wait $!

  find /etc/netplan/ -type f -exec rm {} +
  cat <<EOF >/etc/netplan/custom-network.yaml
network:
  version: 2
  renderer: NetworkManager
  ethernets:
    ${netcard}:  # bridge
      dhcp4: no
      dhcp6: yes
      addresses:
      - ${ip_address}/${subnet_mask_length}
      nameservers:
        addresses: [ "${gateway}", "8.8.8.8" ]
      routes:
      - to: 0.0.0.0/0               # from host to Internet
        via: ${gateway}
    ens37:  # host
      dhcp4: no
      dhcp6: no
      addresses:
      - 192.168.30.100/24
      routes:
      - to: ${subnet_ip}/${subnet_mask_length}  # from host to bridge
        via: 192.168.30.1
      - to: 192.168.30.0/24
        via: 192.168.30.1
EOF
  chmod 600 /etc/netplan/custom-network.yaml
}

enableNetworkConfig() {
  pid1="$(pgrep NetworkManager)"
  netplan apply
  # 检测netplan执行完成
  {
    while true; do
      pid2="$(pgrep NetworkManager)"
      [ "${pid1}" -ne "$pid2" ] && break
      sleep 1
    done
    sleep 1 # 等待新NetworkManager进程生效
  } &
  wait $!

  # dns cache
  {
    resolvectl dns ens37 --cache=yes # host
    if [ "${mode}" = "bridge" ]; then
      resolvectl dns ens33 --cache=yes
      resolvectl dns ens38 --cache=no
      ifconfig ens38 down 
    elif [ "${mode}" = "nat" ]; then
      resolvectl dns ens33 --cache=no
      resolvectl dns ens38 --cache=yes
      ifconfig ens33 down 
    fi
    resolvectl flush-caches
  } &
  wait $!

  echo "----配置完成，开始测试--"
  echo "测试dns"
  resolvectl query baidu.com &
  wait $!

  echo "------测试ping-------"
  ping baidu.com -c 4
  echo "测试完成"
}

getSubnetIP         # 获取子网ip
configNetwork       # 写入配置
enableNetworkConfig # 启用网络配置

 自此，脚本编写完毕，上述脚本中，由于netplan apply指令执行完，但后台会重启NetworkManager，这里通过监听进程号是否改变来判断网络配置应用成功.

  1. 测试结果

  • host+bridge测试结果

     执行sudo bash update-net.sh --mode bridge --net 192.168.31.100/24 --gw 192.168.31.1;

  • host+nat测试结果

     执行bash update-net.sh --mode nat --net 192.168.255.100/24 --gw 192.168.255.1;

     由于宿主机开启了clash的tun模式，自动生成了clash网卡，网卡网段为192.18.0.0/16，因此解析baidu.com时，将tun网卡的网段的随机ip作为fakeip返回，因此resolvectl query baidu.com解析出的地址并不是真正的ip;

注意:
虚拟机重启后仍然需要运行该脚本，可创建一个service用于重启，但仅适用于host+nat模式;

而host+bridge模式可能会由于网段改变而无法固定，最好手动执行该脚本

代理软件

  1. vscode远程开发

 前不久在尝试VScode远程开发(通过ssh remote插件)时，由于需要为远程机器安装插件，但出现报错XHR failed;

 网上的解决方案主要有两种: 1.先离线下载，在通过sftp上传到远程，远程再离线安装; 2.为vscode虚拟机配置http_proxy;

 离线安装的方法未免太过麻烦，配置http_proxy的方式也因为开启tun模式而无法使用;

 后来在clash的日志中，发现当刷新VScode的插件列表时，clash的日志同步出现marketplace.visualstudio.com域名，在clash配置中添加该域名直连规则，并没有用，可能是配置了黑名单模式，可参考该仓库;

 后续查阅相关issue和clash的文档，最后配置规则为PROCESS-PATH, X:\path\to\Code.exe, DIRECT，成功为远程vscode安装插件.

  1. 关于dns泄露

 在之前写的一篇文章中，本人提到了dns泄露等问题，并且也推荐了相关科普视频;

 最近发生的关于承德的新闻，令人震惊，由于B站和微博都有很多相关评价，这里就不展开了;

 在过去有多人就通过跨境电商或者搬运youtube视频牟利，这件事以后，越来越多人会注意隐藏自己，接下来要说的是关于dns泄露;

 目前clash已经更新了好几次，最开始的使用是参考着文档编写配置文件，另外也比较关注相关的安全的问题(包括不安全证书、dns泄露)，针对前者需要对软件安装包进行校验，尽量从安全的链接下载并校验，针对后者尽量采用dot或doh;

 此前尽管注意到了dns泄露问题，但也就在这次新闻后，有必要采用dot或doh，测试泄露可通过该网站(https://surfshark.com/dns-leak-test)进行测试，注意需要在clash中修改mixin，参考配置如下:

mixin: # object
  dns:
    enable: true
    listen: 0.0.0.0:53
    enhanced-mode: fake-ip
    use-hosts: true
    nameserver:
    - tls://doh.pub
    - tls://dns.alidns.com
    # - tls://8.8.8.8       # 谷歌的dot或doh可能屏蔽了
    #- 'tls://1.1.1.1'      # cf的同样被屏蔽
    fallback:
    - tls://1dot1dot1dot1.cloudflare-dns.com  # 未屏蔽，但建议使用自建dns server
    fallback-filter:
        geoip: false
    ipv6: false
    fake-ip-range: 198.18.0.1/16

 最近两天使用代理出了点问题，恰好使用了其他网络软件(n2n，异地组网的软件)，**tun模式卡死(卸载tap网卡、重装service mode、重装CFW和重启系统都不行)**，查看防火墙配置，如下图所示:

 可以发现出现了多个clash虚拟网卡，正常情况下应该只有一个clash网卡，另外clash日志，出现较多的报错如下图:

 可能由于上述配置经常使用1.1.1.1:853和8.8.8.8:853通信请求频繁，然后导致自己所在地区限制了该ip上的853端口，导致所有dns解析失败，

 至于为什么出现多个网卡，可能由于接管了所有dns解析失败，使代理处于disconnected状态，然后反复尝试重连，从而导致旧网卡未删除有创建了新网卡，但在网络连接界面中并没有出现多个clash网卡，具体原因未知;

 相关问题可参考该issue: 开启tun模式后，客户端疯狂对一个ip进行连接，直到clash卡死 ;

&emsp如果出现”wifi已连接”，但”clash无法联网/无网络访问权限”，建议按照以下步骤排除:

  1. 检查防火墙，允许通过clash-win64.exe和clash for windows.exe，一般第一次联网时会请求相关权限，全部允许，或者手动添加;

  2. 检查dns相关配置，可通过软件自带的dns query功能，测试能否解析成功，也就是说在nameserver和fallback中至少有一个能解析成功;

  3. 关于fake-ip-filter，Windows内置dns检测网络状态需要真实ip，可以参考tun网卡默认配置;

  4. 重新配置路由(原路由可能产生了环回)，卸载tap网卡并关闭service mode，然后在win下的cmd中以管理员运行以下指令:

    netsh int ip reset
netsh winsock reset
ipconfig /flushdns

     接着重启电脑生效，此时再打开tun模式，如果需要用到其他网卡，可通过route指令再管理员模式添加路由(务必通过if参数来指定网卡, eg. route add 192.168.30.0 mask 255.255.255.0 192.168.30.1 metric 3 if 8，关于if 8参数可参考该文章).

 该问题的解决耗费了差不多一个星期，整个过程经历了非常多的转折，有时可连上网络但却发现问题才刚刚开始，如此反复，到了10月26日几乎要放弃了，但是由于要访问被墙的服务器，忍不住点了下tun模式，发现竟然好了，非常顺利，此时我并不清楚具体原因，因为26日做了很多尝试(整个上午和半个下午)，不清楚是哪个步骤意外地解决了问题;

 27日上午。直到我要通过host网络连接虚拟机时，才发现竟然无法连通，再尝试bridge网络连接，成功; 开始意识到可能因为host网卡路由和clash网卡路由产生冲突出现环回(可能在使用n2n的vlan后出现的该问题)，导致电脑卡死’

 后来通过cmd执行route add 192.168.30.0 mask 255.255.255.0 192.168.30.1 metric 3 if 8来添加静态路由，然后关闭clash的tun模式，重置网卡路由(命令参考上述的故障排除步骤5)，重启计算机后启动clash的tun模式，成功联网;

 至此，整个问题出现的原因和解决法案都已搞懂，尽管期间磕磕碰碰，经历了好几个转折(21日使用n2n和朋友打游戏，晚上学习时问题开始浮现，逐渐成为接下来近一周的梦魇，22日反复重装网卡和尝试重装软件皆无用,偶然一次成了但重启仍无效，后续两天参考相关issue包括配置允许防火墙、切换gvisor和system模式，24日尝试修改dns配置(nameserver、fallback和fake-ip-filter)，成功一两次，重启仍无用; 26日尝试了前面几天所有的方法，也尝试了修改路由，仍无用，几乎打算考虑采用其他代理软件); 27日上午，偶然间打开tun模式，竟然连通，不断排查原因，最终才知道问题的根本原因在于多网卡环境冲突产生环回(会导致卡死并连不上网络)，dns存在小问题(非主要矛盾，不会造成卡死但会导致连不上网络).

 此外，最近在学习网络编程，了解到quic协议(速度更快)，后续将考虑采用自建dns服务器.

注意: cmd管理员模式重置路由，需要重新配置下VMware网卡包括host网络和nat网络.
