kubernetes学习

 目前的学习已进入到一定门槛，接下来要从单体应用买入到分布式应用;

 尽管此前已采用docker-compose配合shell能够自动构建应用，但也出现了配置管理难、接口调用复杂、扩展性差、安全性差等问题;

 k8s作为目前流行的技术，对以上等问题都做了处理，因此k8s将会成为未来软开必备技能.

 请参考 https://www.bilibili.com/video/BV1MT411x7GH/?p=9&vd_source=7bd360d6355545f91d33a7f79642900c ;

 也可参考官方文档 https://kubernetes.io/zh-cn/docs/concepts/ ;

概述

  1. 历史发展

 传统部署：传统的应用部署直接将应用部署在真实机器上，而由于应用对机器环境与操作系统的要求，因此不具备跨平台能力;另外，如果一台机器运行多个程序，如果某个程序占用大部分资源，则无法确保每个程序有足够的资源，对于物理机的维护非常麻烦;

 虚拟化部署：随着虚拟化技术发展，一台物理机可以虚拟出多台机器，且每个机器是完整的计算机，每个机器是相互隔离的，且与宿主机隔离，提高了安全性;

 容器部署：容器共享操作系统，没有虚拟出整套硬件，比虚拟机更加轻量，资源利用率更高;容器间、容器与宿主机间相互隔离，具备安全性;同一个容器镜像可运行在不同操作系统，具备跨平台能力;

 容器的广泛应用带来了管理与监控困难、接口调用关系复杂、安全性、运行维护、配置管理难、存储挂载管理难等问题，而k8s的出现便为解决这些问题。

Kubernetes对象

 k8s对象: k8s对象是持久化的实体，可描述集群的状态(哪些容器应用在运行、可被使用的资源、运行策略);所有k8s对象包含’对象规约’和’对象状态’两个字段.用于管理对象的配置;创建k8s对象可直接创建或通过kubectl，多数情况需要提供yaml文件描述对象特征;

 对象规约(Spec): 对象创建时必须给定Spec内容，描述希望对象具备的特征，即期望状态(Desired State);

 对象状态(Status): 描述对象的当前状态，由k8s系统和组件设置和更新，任何时刻，k8s控制平面管理对象的实际状态，以达到期望状态;

 给出application/deployment.yaml文件，yaml文件必要字段如下:

apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-deployment
spec:
  selector:
    matchLabels:
      app: nginx
  replicas: 2 # 告知 Deployment 运行 2 个与该模板匹配的 Pod
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx:1.14.2
        ports:
        - containerPort: 80

 通过kubectl apply指令创建对象，kubectl apply -f application/deployment.yaml，输出为:deployment.apps/nginx-deployment created;

 yaml文件必要字段为包括apiVersion(创建该对象所用的k8s api版本)、kind(对象类别)、metadata(对象唯一标识信息，包括name、UID、可选的namespace)、spec(对象的期望状态)等;

Pod对象

 Pod对象: Pod为k8s管理的最小可部署单元，Pod(中文翻译豌豆荚)中有一个或多个容器，这些容器共享资源(存储、网络、运行方式声明);Pod的共享上下文包括一组Linux命名空间、cgroups、隔离容器的技术;

 给出/pods/simple-pod.yaml文件，里面描述运行nginx的Pod，示例如下:

apiVersion: v1
kind: Pod
metadata:
  name: nginx
spec:
  containers:
  - name: nginx
    image: nginx:1.14.2
    ports:
    - containerPort: 80

 通过 kubectl apply -f /pods/simple-pod.yaml 指令创建Pod;Pod具备Init容器和应用容器，Init容器会在启动应用容器之前运行并完成;Pod名称应遵循DNS标签规则;

 Pod操作系统: 对应.spec.os.name字段，值被设为windows或linux，如果宿主机的操作系统与该字段不同，则kubelet会拒绝运行Pod;

 Pod和控制器: 可通过工作负载资源创建和管理多个Pod，可处理副本的管理、上线，提供自愈能力;工作负载资源示例参考: Deploymen、StatefulSet、DaemonSet;

 Pod模板(Pod Template): Pod模板是包含在工作负载对象中的规范，用于创建Pod;修改Pod模板不会对已存在的Pod直接起作用;

 Pod存储: Pod可设置一组共享的存储卷，Pod中容器都可访问该共享卷;

 Pod网络: 每个Pod拥有唯一的IP地址和端口;Pod中每个容器共享网络名称空间，包括IP地址和网络端口，容器间可采用localhost互相通信;

 静态Pod(Static Pod): 直接由特定节点的kubelet守护进程管理，失效时由kubelet重启;

 容器探针(Probe): 用于对容器定期诊断是否正常运行，包含3种动作: ExecAction(借助CRI执行)、TCPSocketAction(kubelet直接检测)、HttpGetAction(kubelet直接监测);

Pod生命周期

 Pod被创建，会被赋予唯一ID(UID)，被调度到节点后在终止或删除前处于运行状态;

 Pod不具备自愈能力，若某节点失效，会在一定时间后删除节点上的Pod，Pod依赖于节点环境与资源;k8s通过控制器来管理Pod对象;

 拥有UID的Pod不会被重新调度到不同节点，而是被新的Pod(名字可不变，但UID会变化)替换;Pod对应的存储卷会随着Pod删除而删除;

 Pod阶段(Pod Phase): Pod阶段是枚举的、有限的，对应phase字段，其取值包括Pending(Pod已被接受但未创建)、Running(Pod已绑定节点且至少一个容器运行，或处于启动和重启状态)、Succeeded(Pod所有容器已终止且不再重启)、Failed(Pod至少一个容器因失败而终止)、Unknown(Pod状态无法获取，通信失败)等;此外，通过kubectl指令展示的Terminating(终止)并不是Pod阶段之一;

 容器状态(Container States): 表示Pod中每个容器的状态，取值包括Waiting(等待)、Running(运行中)、Terminated(已终止);可通过kubectl describe pod <pod名称>来获取Pod每个容器的状态;

 容器重启策略(Container Restart Policy): Pod的spec中包含restartPolicy字段，取值包括Always、OnFailure、Never等，默认值为Always;容器重启延迟分别为10s、20s、40s……，最长为5分钟，一旦容器稳定运行10分钟，重启延迟会重置为10s;

 Pod状态(Pod Conditions): Pod具有PodStatus对象，其包含一个PodConditions数组，PodCondition包括PodScheduled(Pod已被调度到节点)、PodHasNetwork(Pod创建且配置了网络)、ContainerReady(Pod中所有容器已就绪)、Initialized(所有Init容器已成功完成)、Ready(Pod可为请求提供服务，且被添加到负载均衡池中)等;

 Pod就绪态(Pod Readiness): 可通过spec.readinessGates字段列表进行设置，下面给出示例:

kind: Pod
...
spec:
  readinessGates:
    - conditionType: "www.example.com/feature-1"
status:
  conditions:
    - type: Ready # 内置的 Pod 状况
      status: "False"
      lastProbeTime: null
      lastTransitionTime: 2018-01-01T00:00:00Z
    - type: "www.example.com/feature-1" # 额外的 Pod 状况
      status: "False"
      lastProbeTime: null
      lastTransitionTime: 2018-01-01T00:00:00Z
  containerStatuses:
    - containerID: docker://abcd...
      ready: true
...

 Pod就绪态的状态(Status For Pod Readiness): 如果需要设置Pod的status.conditions，可通过kubectl patch指令;Pod被认为就绪需要具备条件: 1.Pod中所有容器都已就绪，2.readinessGates中所有状态都为True;

 Pod网络就绪(Pod Network Readiness): Pod被调度到节点后，需被kubelet接受并挂载卷，完成后kubelet与CRI(Container Runtime Interface)为Pod配置网络，若启用PodHasNetworkCondition，kubelet会通过Pod的status.conditions字段的PodHasNetwork判断Pod是否符合初始化条件;当Pod的沙箱创建和网络配置后，kubelet将PodHasNetwork设为True，然后开始拉取镜像和创建容器;

 Pod初始化(Pod Initialized): 对于带有Init容器的Pod，kubelet会在Init容器成功完成后将Initialized设为True(发生在CRI成功创建沙箱和配置网络之后);若Pod没有Init容器，则在创建沙箱和配置网络前kubelet将Initialized设为True;

 Pod调度就绪态(Pod Scheduling Readiness): 参考Pod调度就绪态;

 容器探针(Container Probes): 用于对容器定期诊断，可通过容器内执行代码或发出网络请求来执行诊断;诊断机制包括exec(容器内执行指令，推出返回码为0即成功)、grpc(目标应事先grpc检查，响应为SERVING表示成功)、httpGet(向目标容器ip:port发送get请求，响应状态码大于等于200且小于400即成功)、tcpSocket(若ip
上端口打开即成功)等;探测结果包括Success、Failure、Unknown等;

 探测类型: 1.livenessProbe(容器是否在运行，若失败则容器被杀死，默认Success)，2.readinessProbe(容器是否准备好提供服务，若失败则该endpoints控制器会删除服务里表中该Pod的ip)，3.startupProbe(容器应用是否已启动，若提供startupProbe，其他Probe会被禁用，若探测失败，容器会被杀死并重启);

 Pod终止(Termination of Pods): 终止Pod不应通过kill信号或指令，会导致其无法完成清理操作;
