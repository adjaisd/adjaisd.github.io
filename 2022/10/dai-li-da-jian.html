代理搭建

 继上次vps被墙后，结合网上教程，总算实现最低需求了，除了延迟高，大多数问题考虑到了;

 如果仍不太明白，可以参考https://bulianglin.com/

环境准备

 服务器系统为ubuntu22.04LTS，需要装nginx、docker、docker-compose和acme.sh;

 服务端采用xray，客户端为clash.

 方案采用cdn+vmess+ws+tls，需要申请域名并配置nameserver为cloudflare的.

问题

证书申请

 acme.sh可能由于网络原因，国内vps采用standalone方式不行，但国外的vps可以;无法解决的话，尝试dns方式，添加一条txt的记录;

 参考https://cloud.tencent.com/developer/article/1134944.

多域名或子域名

 有时vps已部署有域名，但需要新增域名，如果都属于它的子域名，最好不要采用泛域名证书，而是单域名证书，并配置nginx.conf，参考如下:

configuration
-------nginx.conf配置---------
user  root;
worker_processes  auto;
error_log  /var/log/nginx/error.log notice;
pid        /var/run/nginx.pid;
stream {
    map $ssl_preread_server_name $name {
        adja.com.cn       blog; # vmess + ws + tls
        app.adja.com.cn   web;  # 后端web站点
    }
    upstream blog {
        server 127.0.0.1:4443;
        server 127.0.0.1:14443;
    }
    upstream web {
        server 127.0.0.1:4444;
        server 127.0.0.1:14444;
    }
    server {
        listen 443 reuseport;
        proxy_pass $name;
        ssl_preread on;
        proxy_protocol on;
    }
}
events {
    worker_connections  1024;
}
http {
    proxy_max_temp_file_size 5m;
    proxy_send_timeout 120; # 代理发送超时
    proxy_read_timeout 120; # 代理接收超时
    proxy_headers_hash_max_size 51200;
    proxy_headers_hash_bucket_size 6400;
    sendfile        on;
    keepalive_timeout  65;
    gzip on;
    set_real_ip_from 127.0.0.1; # 获取真实客户IP，不然全是127.0.0.1
    real_ip_header proxy_protocol;
    port_in_redirect off; # 重定向去掉端口号
    include       /etc/nginx/mime.types;
    default_type  application/octet-stream;
    log_format  main  '$remote_addr - $remote_user [$time_local] "$request" '
                      '$status $body_bytes_sent "$http_referer" '
                      '"$http_user_agent" "$http_x_forwarded_for"';
    access_log  /var/log/nginx/access.log  main buffer=32k gzip flush=1m;
    include /etc/nginx/conf.d/*.conf;
}
---------default.conf配置------------
# 禁止通过ip地址访问
server {
    listen 80;
    server_name _;
    return 400;
}
# 允许通过域名访问-自动跳转到https
server {
    listen 80 default_server;
    server_name *.adja.com.cn;
    rewrite ^(.*) https://$server_name$1 permanent;
}
-------adja.com.cn.conf配置---------
server {
    listen       127.0.0.1:4443  proxy_protocol ssl;
    listen       127.0.0.1:14443 proxy_protocol ssl http2;
    server_name  adja.com.cn www.adja.com.cn;
    ssl_certificate     /etc/nginx/certs/${server_name}/${server_name}.crt;
    ssl_certificate_key /etc/nginx/certs/${server_name}/${server_name}.key;
    # 博客站点
    location / {
        proxy_ssl_server_name on;
        proxy_pass https://adjaisd.github.io/;
        sub_filter 'adjaisd.github.io' '$server_name';
        sub_filter_once off;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_http_version 1.1;
    }
    location /xray {
        proxy_redirect off;
        proxy_pass http://127.0.0.1:20001;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    }
}
-------app.adja.com.cn.conf配置-------
server {
    listen       127.0.0.1:4444  proxy_protocol ssl;
    listen       127.0.0.1:14444 proxy_protocol ssl http2;
    server_name  app.adja.com.cn;
    ssl_certificate     /etc/nginx/certs/${server_name}/${server_name}.crt;
    ssl_certificate_key /etc/nginx/certs/${server_name}/${server_name}.key;
    # web站点
    location / {
        proxy_pass http://127.0.0.1:8080/;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_set_header cookie $http_cookie;
        proxy_set_header Proxy-Connection "";
    }

 建议测试时，关掉cf的cdn，采用Edge浏览器每次访问清除所有数据包括证书文件，查看证书是否符合访问的域名;

 cdn只会显示泛域名证书，缓存证书可能导致访问非目标域名.

 后续尝试采用vless的回落机制，nginx仍然作为前置，xray作为后端，始终无法访问回落伪装的网站，即便去掉了cdn代理，待解决;

DNS泄露

 访问被墙网站前，本地设备发出的http请求被clash拦截，根据规则决定直连还是代理，在这过程中，当域名需要解析成ip与规则对比时，这时会发出dns请求，如果没有配置dns，那么请求会发到本机配置的dns服务器上即路由器，接着传到路由器配置的dns服务器，一般是运营商的dns服务器，这样运营商就知道你在访问谷歌;

 比较好的方法是，在所有需要将域名解析成ip时以比较rule，在后面加上no-resolve来跳过解析，最后域名按照黑名单模式被block，或者白名单模式解析和响应全由代理负责;

 因此，先把需匹配域名的规则放在前面，匹配ip的规则放后面并且是no-resolve，最后需要个兜底的，当有匹配或都不匹配时对请求block或proxy;

 有时即便客户端配置了dns，也需要注意服务端的dns配置，尽可能配成国外dns服务器，对访问国内的请求尽可能block，减少访问国内vps以减少被墙概率;

 可以参考https://www.youtube.com/watch?v=aKlH6KRt9Jc&feature=youtu.be

配置转换

 代理客户端有很多，有时客户端暂不支持某协议，需要暂用其它客户端，不同客户端配置文件还是有差别的，此时便需要配置转换工具;

 虽然网上有很多配置转换工具，但无法确保转换过程中它有没有记录私人链接，于是最好的方法是采用开源项目在本地搭建类似服务，本地进行转换;

 项目地址：https://github.com/tindy2013/subconverter

 有时订阅链接出现127.0.0.1，而不是合理的域名，需要修改pref.toml的managed_config_prefix字段.

Linux下作为全局代理

 需求: 国内vps需要访问github下载发行的新版docker-compose，如果国内直连，速度会非常慢，常用的方法是在本地下好再上传到vps，或者配置http_proxy或https_proxy，通过这种方法不仅麻烦，而且并不能使用所有场景，如果采用的是udp或icmp等协议，那么常规方法便失效了;

 常用方案是通过iptables转发流量到代理端口，由于对Linux网络配置这块不熟，折腾了几次始终没成功，后续再完善…
