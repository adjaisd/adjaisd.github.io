Jenkins自动构建首个应用

 最近，参加到团队一个开源团队项目，考虑到后期开发与发布效率问题，尝试学习使用Jenkins来自动化构建项目;

 其实在先前搭建hexo的评论系统时采用了vercel的服务，可以前往vercel.com体验下自动化构建;之所以不使用vercel，考虑后续团队公司可能都用jenkins，借助其它工具安全性得不到保证.

 采用docker构建Jenkins，由于涉及敏感信息，彩用ssl来保证安全性.

准备工作

 系统centos7;考虑到jenkins内部的自动安装不太方便，故采用手动配置的方式装须要的依赖，这里的jdk和Maven都安装在本地vps，再挂载到容器内部;

宿主机上准备工作

  1. 安装jdk1.8

 注意那个配置环境变量的cat命令须要单独输入，直接执行可能会有问题

rm /tmp/jdk*.tar.gz -f
wget https://repo.huaweicloud.com/java/jdk/8u202-b08/jdk-8u202-linux-x64.tar.gz -P /tmp
ls -lht /tmp/ | grep jdk
mkdir -p /usr/local/java/
tar -zxf /tmp/jdk-8u202-linux-x64.tar.gz -C /usr/local/java/
# 添加环境jdk1.8环境变量
cat > /etc/profile.d/java.sh <<EOF
export JAVA_HOME=/usr/local/java/jdk1.8.0_202
export JRE_HOME=\${JAVA_HOME}/jre
export CLASSPATH=.:\${JAVA_HOME}/lib:\${JRE_HOME}/lib
export PATH=\${JAVA_HOME}/bin:\$PATH
EOF

source /etc/profile.d/java.sh
ln -s ${JAVA_HOME}/bin/java /usr/bin/java
java -version
  1. 安装maven
# centos安装Maven
rm /tmp/*maven* -f
#wget https://dlcdn.apache.org/maven/maven-3/3.9.0/binaries/apache-maven-3.9.0-bin.tar.gz -P /tmp
wget https://mirrors.aliyun.com/apache/maven/maven-3/3.9.0/binaries/apache-maven-3.9.0-bin.tar.gz -P /tmp
ls -lht /tmp/ | grep maven
mkdir -p /usr/local/maven/
tar -zxf /tmp/apache-maven-*.tar.gz -C /usr/local/maven/
# 添加环境Maven环境变量
cat > /etc/profile.d/maven.sh << EOF
export M2_HOME=/usr/local/maven/apache-maven-3.9.0
export MAVEN_HOME=/usr/local/maven/apache-maven-3.9.0
export PATH=\${MAVEN_HOME}/bin:\${PATH}
EOF 

chmod +x /etc/profile.d/maven.sh
source /etc/profile.d/maven.sh
mvn -version

注意: 如果PATH出现重复的路径，执行以下命令

cat <<EOF >> /etc/profile
export PATH=\$(echo \$PATH | tr : "\n"| sort | uniq | tr "\n" :)

  1. 安装docker-compose
#curl -L "https://github.com/docker/compose/releases/download/1.24.0/docker-compose-$(uname -s)-$(uname -m)" -o /usr/local/bin/docker-compose

docker-compose构建jenkins+nginx

 我的docker-compose如下所示：

version: '3'
services:                                      # 集合
  docker_jenkins:
    user: root                                 # 为了避免一些权限问题 在这我使用了root
    restart: unless-stopped                    # 重启方式
    image: jenkins/jenkins:lts-centos7-jdk8    # 指定服务所使用的镜像 在这里我选择了 LTS (长期支持)
    container_name: jenkins                    # 容器名称
    networks:
      - nginx_net
    ports:                                     # 对外暴露的端口定义
      - "8080:8080"
      - "50000:50000"
    volumes:                                   # 卷挂载路径
      - /home/docker/jenkins/:/var/jenkins_home # 这是我们一开始创建的目录挂载到容器内的jenkins_home目录
      - /var/run/docker.sock:/var/run/docker.sock
      - /usr/bin/docker:/usr/bin/docker        # 这是为了我们可以在容器内使用docker命令
      - /usr/local/bin/docker-compose:/usr/local/bin/docker-compose
      - $JAVA_HOME:$JAVA_HOME
      - $MAVEN_HOME:$MAVEN_HOME
    environment:
      TZ: Asia/Shanghai
      JENKINS_OPTS: --prefix=/jenkins
  docker_nginx:
    restart: unless-stopped
    image: nginx
    container_name: nginx
    networks:
      - nginx_net
    ports:
      - "80:80"
      - "443:443"
    volumes:
      - /home/docker/nginx/conf.d:/etc/nginx/conf.d       # 配置目录
      - /home/docker/nginx/log:/var/log/nginx             # 日志
      - /home/docker/nginx/certs:/etc/nginx/certs         # 证书
      - /home/docker/nginx/html:/usr/share/nginx/html     # 页面，用于后续申请新ssl证书或其他作用
    environment:
      TZ: Asia/Shanghai
networks:
  nginx_net:
    name: my_net
    ipam:
      driver: bridge
      config:
        - subnet: 172.30.0.0/16

 然后执行docker-compose up -d进行构建;

申请ip型ssl证书

 尽管买了域名，但vps在国内，国内备案的话，比较麻烦而且严格，本人采用腾讯云，备案须要vps和定位在同一个地区，比如vps是北京的，但本人在江苏那么就不行。即便备案成功了光审核也要好久，类似的其它云服务供应商应该差不多情况;

 如果买的不是国外vps的话，那么ip型域名证书也算是一种应急;本人用过zerossl，有些操作可能要用到梯子;

  1. 采用zerossl手动申请;

 通过文件认证域名(可以是公网ip)，如果采用上面的docker-compose构建nginx的话，将认证文件放到挂载目录/home/docker/nginx/html下，并配置nginx，如下:

# 我的请求地址为http://{ip地址}/well-known/pki-validation/E07978EC02A019B2AE16CA6EBB6487FC.txt
server {
    listen         80;
    listen    [::]:80;
    # ssl证书盛情验证归属文件
    location /well-known/pki-validation/ {
        root /usr/share/nginx/html;
        rewrite ^/well-known/pki-validation/(.*)$ /$1 break;
    }
}

 认证完成后，下载nginx形式的证书文件，将ca_bundle.crt内容复制到certificate.crt文件末尾，保留certificate.crt和private.key文件.

  1. 采用acme.sh脚本自动申请和续期ssl证书;

 参考官网https://github.com/acmesh-official/acme.sh

nginx配置https访问Jenkins

 在完成域名验证后，将key和crt证书文件上传到/home/docker/nginx/certs/ip_ssl目录下，然后参考下面nginx配置:

server {
    listen       443 ssl http2 default_server;
    server_name  _;

    # ip_ssl证书文件
    ssl_certificate     /etc/nginx/certs/ip_ssl/certificate.crt;
    ssl_certificate_key /etc/nginx/certs/ip_ssl/private.key;

    ssl_session_timeout 10m;
    ssl_protocols TLSv1 TLSv1.1 TLSv1.2;
    ssl_ciphers ECDHE-RSA-AES128-GCM-SHA256:HIGH:!aNULL:!MD5:!RC4:!DHE;
    ssl_prefer_server_ciphers on;
    ssl_session_cache shared:SSL:10m;
    # error_page 497  https://$host$request_uri;

    error_page   500 502 503 504  /50x.html;
    location = /50x.html {
        root   /usr/share/nginx/html;
    }
    # jenkins访问路径
    location /jenkins/ {
        proxy_set_header X-Rea $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header Host $http_host;
        proxy_set_header X-Nginx-Proxy true;
        proxy_pass http://jenkins:8080;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}

 记得重启nginx容器;

 在新版Jenkins即jenkins/jenkins:lts中使用gitee插件毫无问题，但在旧版，发现即便添加了gitee的token，但在配置Maven的Job时发现gitee的token拉去不下来，意味着无法拉取仓库代码;

远程vps自动构建镜像

 大概过程是，先从源码仓库通过渠道(ssh公钥或token)克隆代码，再自动执行打包命令，将代码变为Jar包，然后再通过ssh或者代理将jar包发布到远程并执行自定义构建指令;

 如果自己手写脚本，需要非常熟悉shell脚本、maven指令与git指令还有配置，但通过jenkins可以很容易实现自动化构建，而且扩展性很高，可以邮件提醒，多元程发布等;

 实现自动化构建需要安装插件，Maven Integration plugin与Publish Over SSH，前者允许创建Maven项目的Job，后者用于通过ssh发布到远程服务器，只需要配置publish over ssh;

准备工作

  1. 配置publish over ssh

 请参考该视频https://www.bilibili.com/video/BV1bS4y1471A/?p=11&vd_source=55b0017bcef56ebdd1401ec0fe6fe6b7

 在Dashboard –> 系统管理 –> 系统配置 –> 找到publish-over-ssh，如下所示:

 在ssh-server上面的是默认密码或私钥，在下面远程ssh没有配密码或私钥时会使用默认的密码或私钥;

 在ssh-server中，hostname为目标vps的ip，username为目标服务器的用户，高级设置为该用户对应的密码或私钥;关于密码或私钥配置，参考说明:

Override the default authentication key.
The key is specified by pasting into the Key section, or by providing the file path to the key in the Path to key. If both are provided, then the pasted Key will take precedence.
If no key is set then the Passphrase / Password will be used for password authentication, else it is used for decrypting the key.
---
覆盖默认身份验证密钥。
通过粘贴到密钥部分或通过在密钥路径中提供密钥的文件路径来指定密钥。 如果两者都提供，则粘贴的密钥优先。
如果未设置密钥，则密码短语/密码将用于密码验证，否则将用于解密密钥。

开始构建

 在Dashboard –> 创建任务 –> 输入任务名字(必填),选择Maven项目，点击确认;

 配置源码管理，当修改Repository Url时，Jenkins会测试是否联通，如下所示:

 由于初次构建，其他不用管保持默认，接着修改Post Steps，用于发布到远程vps并执行指令，添加Send files or execute commands over SSH;

 由于是从容器发布到宿主机，传输文件就不需要了，因为试过了传输jar包每次都导致网络卡顿，再加上已经挂载工作目录，这里只利用到远程执行脚本构建镜像，脚本如下:

pro_name=test # 项目名称
tmp_space=/home/docker/jenkins/workspace  # 宿主机挂载jenkins目录的绝对路径
src=${tmp_space}/${pro_name}
dst=/home/workspace/app/${pro_name} # 目标路径可以修改成自定义的
echo "清理目标路径" && rm ${dst} -rf
mkdir -p ${dst} && cd ${dst}
echo "-------开始拷贝-------"
cp ${src}/*Dockerfile ${src}/*docker-compose.yml ./
mkdir ./target -p && cp ${src}/target/*.jar  ./target
echo "-------拷贝结束-------"

res=`docker images | grep app | wc -l`  # 判断镜像是否存在
if [ $res -eq 1 ]; then
        echo "存在镜像: $(docker images | grep app)"
        docker stop app && docker rm app && docker rmi app
fi
echo "-----重新构建-----"
docker-compose up -d
echo "-----构建结束-----"

 由于远程仓库已经上传了Dockerfile与docker-compose.yml文件，这里也展示下吧;

 显示Dockerfile;

FROM openjdk:8-jdk
WORKDIR /root

ENV TZ=Asia/Shanghai
ENV JAVA_OPTS="-Xmx800m"
RUN ln -snf /usr/share/zoneinfo/$TZ /etc/localtime && echo $TZ > /etc/timezone

# 拷贝jar包到镜像
COPY target/*.jar /root/app.jar

# RUN_MODEL 为docker-compose中定义的参数
CMD java -jar ${JAVA_OPTS} app.jar ${RUN_MODEL} --server.port=${PORT}

 然后docker-compose.yml

version: '3'
services:
  docker_app:
    build:
      context: .
      dockerfile: Dockerfile
    image: app
    container_name: app
    networks:
      - jenkinsnginx_app_net  # 网络名称需要自己修改
    ports:
      - "8082:8080"
    environment:
      TZ: Asia/Shanghai
      RUN_MODEL: --spring.profiles.active=default
      PORT: 8080
networks:
  jenkinsnginx_app_net: # 网络名称需要自己修改，替换成nginx和jenkins所在网络
    external: true

 上面的Dockerfile和docker-compose.yml最好上传到远程仓库，方便运维快速构建;

 最后，配置nginx反向代理，转发请求到项目所在端口.

总结

 通过这次实验，综合实践了docker+nginx+jenkins，能够实现https访问ip型jenkins后台;

 挑战虽多，未完整系统学习过nginx，没直接采用最稳妥方案搭建，以致遇到很多少见的问题，比如jenkins版本不兼容jdk1.8，自签名证书，为公网ip而非域名申请ssl证书，直接挂载vps的jdk和maven到容器是否会无法执行(容器不一定为centos7)等;

 意识到安全的重要性，尤其在Jenkins后台，要用到token(权限粒度粗)和远程vps的私钥等敏感信息，那么使用ssl就有必要了;zerossl的证书有效期只有9个月，后期考虑采用acme.sh脚本自动申请和部署证书.

 从零开始实现项目自动化构建并部署，有的没的坑都走过.
